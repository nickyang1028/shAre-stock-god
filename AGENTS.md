## AI交互准则

### 澄清优先原则

当需求、目标、边界或验收标准不清楚时，必须先向用户提问，等待用户确认后再开始开发。

以下情况必须暂停并询问用户：

- 不确定用户真正想实现的是哪个功能或业务流程。
- 存在多种实现方案，且不同方案会影响架构、数据结构、用户体验或后续维护。
- 需要新增依赖、引入框架、修改目录结构、调整 API 契约或改变已有模块边界。
- 需求涉及交易信号、行情数据、指标算法、回测逻辑等容易产生业务误解的内容。
- 需要删除、重命名、迁移或大规模重构现有文件。
- 缺少必要上下文，继续开发只能依赖猜测。
- 用户给出的需求和当前代码、文档或已有约定冲突。

提问时应遵守：

- 一次只问最关键的 1 到 3 个问题。
- 优先给出推荐选项，并说明取舍。
- 不要在目标不清晰时直接编写代码。
- 如果可以做出低风险默认假设，必须先明确说明假设，再继续。

在用户确认前，只允许进行只读探索，例如查看文件、梳理现有结构、总结可选方案，不允许修改业务代码。

### 事实依据原则

回答或开发前必须优先读取项目中的真实文件，不要凭空假设项目结构、依赖、脚本或 API。

- 不确定某个库、函数、类型或接口是否存在时，先搜索代码。
- 不确定运行命令时，先查看 `package.json`、配置文件和现有脚本。
- 不要编造不存在的文件路径、组件、接口、环境变量或测试命令。
- 如果信息无法从代码中确认，必须明确说明“当前仓库中未找到依据”。
- 
### 最小改动原则

实现需求时保持改动范围尽量小。

- 优先沿用现有架构、命名、代码风格和工具链。
- 不做与当前需求无关的重构。
- 不随意新增抽象、依赖或复杂设计。
- 修改共享类型、API 契约或目录结构前，必须说明影响范围。

## 通用编码规范


- ts以及tsx文件中的所有数据类型都要明确定义，禁止出现任何以any标注的变量类型。
- 判断数据类型时，优先使用 `utils` 中已有的方法，例如 `isNumber`、`isString`、`isPlainObject`、`isFunction`、`isThenable`、`isPrimitive`、`isNonNullable`。
- 仅当 `utils` 中没有合适方法，或当前场景需要更严格、更特殊的判断逻辑时，再使用内联 `typeof`、`instanceof` 等判断方式。


## 注释规范（强制）

为了保证代码的可读性与可维护性，所有代码必须遵循以下注释规范：

### 函数 / 组件定义注释（必须）

所有函数定义、React 组件定义前，必须添加 JSDoc 风格注释，格式如下：

```ts
/**
 * 函数 / 组件的一句话描述
 * @param {Type} paramName 参数说明（多个参数逐行列出）
 * @returns {Type} 返回值说明
 */
```

### 函数/组件内部注释（必须）

在函数或组件实现内部，必须根据逻辑复杂度添加必要注释，包括但不限于：

- 关键业务逻辑说明
- 复杂条件判断解释
- 边界情况处理
- 副作用说明（如请求、状态变更）
